如果你在回答的时候，只会说"我把用户的话重写一遍"，那面试官大概率会觉得你根本没做过真实的工程项目。

今天我就带大家深度拆解一下Query改写的底层逻辑。咱们直接看干货。

## 节目开场

大家好，我是彭宇。

## 第一章：为什么要改写Query？三个核心理由

### 理由一：解决语义鸿沟

用户提问说的是大白话，比如"那个啥怎么弄"。

而你的知识库里全是硬邦邦的专业文档，比如"关于某某操作的规范流程"。

**这两者在向量空间里离的可能得有十万八千里。**

如果你直接拿原话去搜，大概率搜出来一堆废纸。

**改写就是为了把口语对齐到专业书面语上。**

### 理由二：消除意图模糊

很多时候用户是很懒的，他们的提问往往缺失背景。

比如他直接问"怎么优化"，如果没有上下文，模型根本不知道：
- 是在问优化代码？
- 优化模型？
- 还是优化流程？

**改写的作用就是通过引入背景信息，把模糊的提问补充成一个具体的、可检索的、声明式描述。**

### 理由三：关联历史信息（最常见）

在多轮对话里，用户特别喜欢说"他那个"、"刚才说的那个"。

**如果没有改写器去执行指代消解，把这些代词替换成真实的实体，那你的检索系统就会像失忆了一样。**

对话进行到第三轮就会壁崩。

### 总结：改写的本质

**改写其实是给检索系统装上了记忆和理解力。**

## 第二章：Query改写的三条核心路子

明白了动机，那具体怎么改呢？我给大家总结了三条核心路子。

### 路子一：语义增强型（HYDE）

**算法名称：HYDE**

**逻辑很直接：**
让模型先针对问题写个假设性的答案。

然后我们不用原始问题去搜，而是用这个假答案去库里搜。

**为什么有效？**

因为假答案跟知识库里的真答案在风格上更接近。

**这在技术上叫"以答案搜答案"**——召回率能直接拉满。

**流程：**
```
用户问题
  ↓
生成假设答案
  ↓
用假答案去知识库搜索
  ↓
获得更相关的结果
```

### 路子二：任务分解型（Step Back / 子查询）

如果用户问"对比一下A公司和B公司的财报"，模型往往会顾此失彼。

这时候咱们就得用**Step Back**或者**子查询拆解**。

**操作方式：**

把一个大坑拆成两个小坑：
1. 先搜A财报
2. 再搜B财报
3. 分头行动
4. 最后再汇总

**这就叫"化整为零"**——是解决复杂问题的必经之路。

**流程：**
```
复杂查询（对比A和B）
  ↓
拆解成子查询
  ├─ 查询A的财报
  └─ 查询B的财报
  ↓
并行检索
  ↓
合并结果
  ↓
最终对比分析
```

### 路子三：上下文补全型（多轮对话）

这就是咱们刚才说的，专门对付多轮对话。

**功能：**

负责把对话历史里的关键信息脱水提取出来，把模糊的具体点翻译成完整的检索需求。

**确保每一轮检索都具备独立的语义。**

**流程：**
```
第一轮对话
  ↓
第二轮对话（含代词）
  ↓
补全上下文
  ↓
生成完整的检索Query
  ↓
检索
```

## 第三章：工程实战中的三个核心难点

讲到这儿，肯定有同学觉得"这不挺简单吗？写个prompt让模型重写一下不就行了"。

**哎，大家注意——真实的工程实战里全是坑。**

咱们重点看这三个核心难点。

### 难点一：延迟（Latency）

**问题：**

你要改写就得额外调用一次大模型，用户可能就得盯着屏幕多等2秒钟。

**两秒钟在互联网产品里可是致命的。**

**解决方案：**

1. **小模型改写**：用经过微调的小模型专项负责改写
   - 速度快
   - 成本低

2. **并行执行**：让改写任务跟初筛检索并行执行
   - 用异步流水线来压缩时间
   - 隐藏改写的延迟

**架构示意：**
```
用户输入
  ├─ 并行任务1：改写Query
  └─ 并行任务2：初筛检索
  ↓
汇合结果
```

### 难点二：查询漂移（Query Drift）

**问题：**

有时候模型会自作聪明，改着改着就把意思改偏了。

**例子：**
- 用户问："苹果怎么吃"
- 模型改写成："苹果手机的售后维修"

**后果：** 检索结果完全离题。

**解决方案：**

引入**语义相似度检查**（余弦相似度、Jaccard相似度等）。

如果改写后的语义跟原话跑偏太多，咱宁可信原话，绝对不能被带节奏。

**流程：**
```
原始Query
  ↓
改写Query
  ↓
计算语义相似度
  ↓
相似度 > 阈值 ✓ 使用改写Query
相似度 < 阈值 ✗ 使用原始Query
```

### 难点三：成本（Cost）

**问题：**

每一条请求都去改写，那token飞得哗哗的流。成本瞬间爆炸。

**解决方案：**

在架构上加一个**意图路由**（Intent Router）。

**路由策略：**
- **简单提问**（如"打招呼"、"表达感谢"）→ 直接跳过改写
- **复杂模糊的长句子** → 才开启改写

**这叫"好钢用在刀刃上"。**

**架构示意：**
```
用户输入
  ↓
意图判断
  ├─ 简单意图 → 直接检索（不改写）
  └─ 复杂意图 → 改写 → 检索
```

## 第四章：面试通关法则——四步惊艳面试官

说了这么多，如果明天你就去面试，这道题到底怎么回答才最惊艳？

我给你总结了一个**四步通关法**。建议大家记一下笔记。

### 第一步：明动机（Why）

**你要先谈为什么改写是解决以下问题的关键：**
- 语义鸿沟
- 意图模糊
- 历史关联

**为什么这样做能打分：**
这显得你很有大局观，不是为了改写而改写，而是清楚问题的症结所在。

**示例回答：**

> "在RAG系统里，用户的自然语言和知识库的专业文档存在语义鸿沟。直接检索往往不准确。改写的目的是三重的：一是对齐语言风格，二是消除歧义，三是在多轮对话中追踪上下文。"

### 第二步：分维度（How）

**把三个技术路径甩出来：**
1. 语义增强（HYDE）
2. 任务分解（Step Back）
3. 上下文补全

**为什么这样做能打分：**
显出你的专业深度，说明你不是纸上谈兵，而是知道具体怎么做。

**示例回答：**

> "我们的改写方案分三层：
> 1. 对于单一查询，用HYDE方法生成假设答案去搜索。
> 2. 对于复杂查询，用Step Back将其拆解成子查询。
> 3. 对于多轮对话，我们提取历史上下文来补全当前的Query。"

### 第三步：论权衡（Trade-offs）

**主动聊聊你对以下问题的工程思考：**
- 延迟（Latency）
- 漂移（Query Drift）
- 成本（Cost）

**为什么这样做能打分：**
面试官听到这儿会觉得你是有实战底蕴的，不是只懂理论。

**示例回答：**

> "改写带来收益，但也有三个挑战：
> - 延迟：我们用小模型 + 异步并行来压缩时间
> - 漂移：用语义相似度检查来避免偏离原意
> - 成本：用意图路由来智能决定是否改写，简单问题直接跳过"

### 第四步：说闭环（Evaluation）

**提到你会用评估框架去量化改写的效果，而不是拍脑袋觉得好。**

**为什么这样做能打分：**
证明你是以数据驱动决策，这是一流工程师的标志。

**示例回答：**

> "最后，我们用评估框架量化改写效果：
> - 用NDCG、MRR这样的召回指标来衡量检索质量
> - 用人工标注的黄金集合来验证改写是否提升准确率
> - 定期A/B测试来对比改写 vs 不改写的端到端效果"

### 完整回答框架

**如果把四步串起来，一个完整的面试回答长这样：**

---

> **面试官问：** "你的RAG系统是怎么进行Query改写的？"

> **你的回答：**
>
> "好问题。首先说Why——改写的意义在于解决三个问题：语义鸿沟、意图模糊和历史上下文的追踪。
>
> 在How的层面，我们实现了三层改写策略。单一查询用HYDE方法，通过生成假设答案来对齐知识库的语言风格。复杂查询用Step Back拆解成子查询。多轮对话通过上下文补全来消解代词。
>
> 但工程中有三个关键权衡。延迟方面，我们用微调的小模型加异步并行来加速。查询漂移方面，我们加了语义相似度检查来防止偏离原意。成本方面，我们用意图路由来智能决定是否改写。
>
> 最后，我们用NDCG、MRR等评估指标和黄金集合验证来量化改写的效果，而不是凭感觉。"
