# 1. 固定窗口？
## 如何实现？
* key 设计成：`rate_limit:{userId}:{yyyyMMddHHmm}`
* 每次请求对这个 key 执行 `INCR`
* 第一次创建时设置过期时间 `EXPIRE 60`
* 如果计数超过 100，就拒绝
## 优点
实现简单，性能高。

### 缺点

有**临界问题**：  
用户可能在 `12:00:59` 发 100 次，在 `12:01:00` 再发 100 次，瞬间 2 秒发了 200 次。


# 2.  滑动窗口限流
比固定窗口更平滑，线上更常用。

### 思路

例如要求：

- 60 秒内最多 100 次请求
    

做法：

- 用 `ZSET`
    
- score = 当前时间戳（毫秒）
    
- member = 唯一请求 ID
    
- 每次请求时：
    
    1. 删除 60 秒以前的数据
        
    2. 统计当前窗口内请求数
        
    3. 如果数量 < 100，允许并写入新请求
        
    4. 否则拒绝

# 令牌桶
这是工程里非常常见、也更合理的方案。

### 思路

系统按固定速率往桶里放令牌，请求来了先拿令牌：
- 有令牌：放行
- 没令牌：拒绝
可以同时支持：
- 平均速率限制
- 短时突发流量

例如：
- 每秒产生 10 个令牌
- 桶容量 20

那么平时每秒最多处理 10 个请求，但允许短时突发最多 20 个。
### Redis 实现思路

通常要在 Redis 中保存两个值：
- `tokens`：当前令牌数
- `last_refill_time`：上次补充时间

每次请求时：
1. 根据 `当前时间 - 上次补充时间` 计算应该新增多少令牌
2. 令牌数不能超过桶容量
3. 如果令牌 >= 1，则减 1 并放行
4. 否则拒绝

这种逻辑也建议用 **Lua 脚本** 保证原子性。
